The short version is: the graph added more opportunities to be wrong than to be helpful. Your earlier results already showed that the system is highly sensitive to wrong or slightly-off injections: matched examples can flip a task from 0% to 100%, but mismatched ones can also tank strong baselines. You also already proved that bigger models are more prone to interference from augmentation than smaller ones.

So if the flat system stayed about the same when you expanded from 48 to 187 examples, but the graph got worse, especially on larger models, that suggests a few likely things.

First, the graph is probably increasing recall without increasing precision. A flat retriever can accidentally stay conservative: it finds a strong local match or nothing useful. A graph tends to pull in neighbors, dependencies, and related patterns. That sounds smart, but in practice it means more “almost relevant” material gets injected. For code generation, “almost relevant” is often worse than nothing. Your prior benchmarks already showed that unrelated or weakly related augmentors can cause regressions, and your hardening plan explicitly focuses on “do no harm,” higher thresholds, and tighter injection limits for that reason.

Second, larger models are likely being overconstrained. Smaller models benefit more from scaffolding because they have more missing structure to fill in. Larger models often already know multiple workable implementations, so the graph can distort rather than assist. That lines up with your earlier observation that the 0.5B model got the biggest gains from augmentors, while stronger models saw smaller gains or even regressions depending on injection quality.

Third, graph edges may be semantically true but operationally wrong.
Example: an HTTP router may indeed relate to decorators, regex, and dispatch maps. But if the user’s task only really needs decorator plus simple path params, adding extra graph-linked patterns can bias the model toward the wrong implementation shape. The graph is representing conceptual relatedness, but the benchmark rewards the exact executable pattern. That mismatch hurts.

Fourth, 187 examples likely pushed you into retrieval dilution. With a larger library, the graph has more ways to wander into nearby-but-bad territory. Flat search may saturate because top-1 or top-2 matches are still roughly the same as before, so quality stays flat. But graph expansion compounds noise. Your own plan to raise thresholds and reduce injection count under lower confidence is exactly the kind of defense this outcome points toward.

So I would not read this as “pattern graph theory failed.” I would read it as:

Unbounded graph expansion is the wrong control strategy.

The graph may still be useful, but probably not as a direct injector.

What I think is happening is this:

Flat retrieval is acting like a precision tool.
Graph retrieval is acting like a recall tool.
Your system’s success depends far more on precision than recall.

That is why the graph gets worse as the library grows.

What to do next

I would split the graph into a different role.

1. Stop using the graph to add more examples directly

Use the graph to re-rank or validate the top flat matches, not to expand them.

So instead of:

retrieve top flat match
expand through graph
inject 3–5 linked examples

Do:

retrieve top 5 flat matches
use graph only to score which one is structurally coherent with the query
inject only 1, maybe 2 max

That keeps graph information while preserving precision.

2. Make the graph a planner, not an injector

Let it answer:

what subpatterns are likely involved?
what failure families does this task belong to?
what missing capability might need support?

Then inject only the single most decisive concrete example.

So graph says:

parser
tokenization
recursion

But the injector picks only:

recursive descent parser blueprint
3. Add a “harm gate”

You already have enough evidence that bad injection is costly. Build a gate like:

if top score < threshold, inject nothing
if graph-expanded set disagrees too much, inject nothing
if model tier is large, allow only hint-level injection

Your plan already points toward stricter confidence-based injection. This result strengthens that direction.

4. Separate graph edges by type

Not all edges are equal. You probably need at least three classes:

hard dependency: must co-occur
implementation sibling: related, but alternative
semantic neighbor: similar topic only

Only hard dependencies should ever influence injection automatically. The other two should maybe influence ranking, not context.

5. Benchmark graph usefulness outside injection

Test three modes:

flat inject
graph re-rank + single inject
graph plan only + no extra inject

My guess is the third or second will beat direct graph injection.

My best current read

The core lesson is:

Your system is not bottlenecked by access to more patterns. It is bottlenecked by choosing fewer, cleaner, more authoritative patterns.

That also explains why flat stayed stable at 187 examples. The extra library size did not help much, but it also did not hurt much because the flat selector stayed relatively narrow. The graph, by contrast, gave the model too much shaped context.

So the next frontier is probably not:

more examples
deeper graph traversal
richer connectedness

It is:

better abstention
stricter authority
smaller injection payloads
graph-assisted selection rather than graph-assisted expansion
